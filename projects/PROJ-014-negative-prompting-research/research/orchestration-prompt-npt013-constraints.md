## Domain 3: Adversarial Quality Gates

**Domain rationale:** The /adversary skill is the quality enforcement mechanism. Deferring it, bypassing it, concentrating it in one agent, or proceeding below threshold allows defective outputs to propagate through the pipeline.

### AQ-1: Gate Launch and Threshold

**Source raw items:** 6, 7

```xml
<constraint format="NPT-013">
NEVER allow a creator agent to hand off output downstream without first launching /adversary at C4 and confirming the adversarial quality score is >= 0.95 -- Consequence: sub-threshold outputs reach downstream agents, defects compound across phases, and the final deliverable inherits compounded quality debt that requires full-pipeline rework to resolve. Instead: invoke /adversary immediately after each creator agent completes, block the handoff until the score is confirmed >= 0.95, and if the score is below threshold, return the output to the creator with critic findings before proceeding.
</constraint>
```

### AQ-2: Iteration Bound

**Source raw item:** 8

```xml

Instead: declare the maximum iteration count in the Orchestration Plan before execution begins (default ceiling: C4=10 per RT-M-010), activate the circuit breaker when the ceiling is reached, and escalate to the user with the current best result and open critic findings.
</constraint>
```

**Source raw items:** 9, 13

```xml
<constraint format="NPT-013">

Instead: route all critic feedback back to the originating creator agent via a structured handoff containing the specific finding, the affected artifact path, and the required correction, then re-run the adversarial gate on the revised output.

## Domain 6: State and Documentation Integrity

### SI-1: State Persistence

```xml
<constraint format="NPT-013">
NEVER rely on in-context memory to persist work state across turns, and NEVER begin execution on a work item without first creating the corresponding worktracker entity -- Consequence: state retained only in the context window is lost on compaction or session end, making work unreplayable and untraceable; work without a worktracker entity is invisible to orchestration and cannot be quality-gated or handed off. Instead: create the worktracker entity (Spike, Story, Task, or Enabler as appropriate) before the first tool call on any work item, and persist all intermediate state to the entity's artifact files rather than relying on context memory.
</constraint>

Instead: update the worktracker entity status, blocked flags, and artifact paths immediately after each state change (start, block, complete, fail), before any other work proceeds.
</constraint>

### SI-3: Documentation Currency

## Domain 7: Prompt Craft

**Domain rationale:** Prompt quality directly determines agent output quality. Role-play breaks the evaluation frame; human code review defeats the quality automation purpose; positive-only framing degrades behavioral compliance relative to NPT-013 constraints.

### PC-1: Role-Play Prohibition

**Source raw item:** 32

```xml
<constraint format="NPT-013">

Instead: address agent behavior by modifying the agent definition's identity, persona, or guardrails sections directly, and invoke the appropriate specialized agent rather than asking a general agent to simulate specialization.
</constraint>
```

### PC-2: Human Review Bypass

**Source raw item:** 33

```xml
<constraint format="NPT-013">

Instead: ensure all code exits the pipeline having passed automated linting (ruff), type checking (mypy strict), >= 90% test coverage (H-21), and adversarial quality scoring >= 0.95, so that human review is an optional audit rather than a required quality gate.
</constraint>
```
