NEVER execute STORY-018 (governance YAML migration) before STORY-017 (rule file changes) is fully complete and its acceptance criteria are verified -- Consequence: governance YAMLs reference the new tier definitions (T3=Persistent, T4=Persistent+External) before those definitions exist in agent-development-standards.md; schema validation runs against an inconsistent state where YAML values are valid but the rule file still describes the old model; every subsequent story (STORY-019, STORY-020) operates on a split-brain codebase where the tier numbers are migrated but the semantics are not yet authoritative; rollback from this state requires reverting both stories in coordination. </constraint>

<constraint format="NPT-013" id="C-02" label="sed Anchor Bug">
NEVER use an end-of-line `$` anchor in sed substitution commands when migrating tool_tier values in governance YAML files -- Consequence: the sed pattern `s/tool_tier: T3$/tool_tier: T4/` silently skips any line where T3 is followed by whitespace or an inline comment (e.g., `tool_tier: T3  # External`); at least one governance YAML (diataxis-explanation.governance.yaml) carries such a comment; the post-migration count shows 48 instead of the expected 49 T3-to-T4 changes; STORY-020 verification fails with a count discrepancy that has no obvious cause because the sed command reports no errors; debugging requires re-auditing all 89 files manually. Instead: use a word-boundary pattern that anchors to the tier value itself, not the end of line -- e.g., `s/\(tool_tier: \)T3\b/\1T4/` -- and run a dry-run pass with `grep -n 'tool_tier: T3'` across all governance YAMLs before and after migration to confirm counts match expectations.
</constraint>

<constraint format="NPT-013" id="C-03" label="grep False Positive">
NEVER count tier occurrences for pre- or post-migration verification using an imprecise pattern such as `grep -rl 'tool_tier:.*T2'` -- Consequence: at least one governance YAML carries an inline comment containing a prior tier reference (e.g., `# Upgraded from T2`); the imprecise pattern matches the comment as a live T2 declaration; the pre-migration T2 count shows 29 instead of the correct 28; post-migration verification produces a spurious discrepancy where the expected delta does not match; the executor wastes diagnosis time on a non-existent migration error, and the verification step produces a false FAIL that may cause the migration to be unnecessarily repeated or rolled back. Instead: scope grep to the key-value field only, excluding comment text -- use `grep -rn '^\s*tool_tier: T2$\|^\s*tool_tier: T2\s' skills/` or strip comments before counting -- and cross-reference the count against the known tier distribution table from ADR-STORY015-001 (T1=4, T2=28, T3=49, T4=7, T5=1) before treating any discrepancy as a migration failure.
</constraint>

<constraint format="NPT-013" id="C-04" label="Security Over-Scoping">
NEVER treat STORY-020 security review as an attack surface analysis of the governance YAML changes -- Consequence: the actual enforcement boundary for tool access is the `.md` frontmatter `tools` field parsed by the Claude Code runtime, not the governance YAML `tool_tier` value; no runtime tool access changes as a result of this migration; an attack surface analysis of the YAML changes finds no exploitable surface and wastes tokens; meanwhile the actual governance risk -- that the permission ceiling in agent-development-standards.md is being expanded (T3 agents gain MK access; T4 agents gain web access) -- goes unverified; the security story produces a misleading PASS on the wrong artifact.
