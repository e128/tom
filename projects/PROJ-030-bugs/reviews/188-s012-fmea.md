## Summary

FMEA of the seven configuration elements in the #188 Dependabot configuration reveals **2 Critical findings** (RPN >= 200), **8 Major findings** (RPN 80–199), and **12 Minor findings** (RPN < 80), yielding a total RPN of 2,206. The highest-risk element is Element 3 (`allow: dependency-type: direct`) with a combined RPN of 530, driven primarily by the undetectable transitive CVE detection latency (FM-009, RPN 280) and silent security exposure in indirect dependencies (FM-010, RPN 160). The second highest-risk element is Element 7 (absence of `applies-to: security-updates`) with a combined RPN of 476.

The overall configuration is **ACCEPT with targeted corrections**. The grouping architecture, direct-only filter, and security ungrouping decisions are sound. The two Critical findings (FM-009 and FM-019) represent operational gaps rather than architectural flaws — both are addressed by monitoring and documentation controls rather than configuration restructuring. The eight Major findings call for clarifying comments, operational runbook entries, and one concrete configuration addition (security update `open-pull-requests-limit` separation). No findings invalidate the configuration design.

## Finding Details

### FM-009-20260313T001: Transitive CVE Detection Gap (Critical)

| Attribute | Value |
|-----------|-------|
| **Severity** | Critical |
| **Element** | E-03: `allow: dependency-type: direct` |
| **Strategy Step** | Step 2 (Enumerate Failure Modes) + Step 3 (Rate S/O/D) |

**Evidence:**
The `dependabot.yml` D3 comment states: "Transitive dep compromises will NOT generate a Dependabot PR. Compensating DETECTION controls: `pip-audit` in CI catches known CVEs; `uv.lock` diff review on grouped PRs reveals transitive shifts... Maximum transitive CVE detection latency: up to 7 days (next weekly Dependabot run triggers parent bump → CI runs pip-audit)."

The risk analysis (`dependabot-risk-analysis.md`, Section 4) confirms: "The security update individual PR requirement is non-negotiable." The `allow: direct` policy means that a transitive dep (e.g., `gherkin-official`, `attrs`, `rpds-py`, `mako`) with a disclosed CVE will not trigger a Dependabot security PR. The compensating control (pip-audit in CI) runs only when a PR is open — it does not run autonomously on a scheduled basis to catch CVEs in the current lockfile state.

**Analysis:**
Severity=8: A transitive dep CVE undetected for up to 7 days represents a genuine vulnerability window for production use of the Jerry CLI. While Jerry's threat model is lower-risk than a web service, `jsonschema` and `pyyaml` are runtime dependencies with user-facing blast radius. Occurrence=5: Transitive dep CVEs occur regularly in mature ecosystems; the `attrs`, `rpds-py`, and `requests` families have had recent CVEs. Detection=7: The failure mode is difficult to detect without pip-audit running independently of PR activity — there is no autonomous daily or weekly CVE scan of the current lockfile.

**Recommendation:**
Add a scheduled GitHub Actions workflow (e.g., `security-scan.yml`) that runs `uv run pip-audit` on a daily or weekly schedule against the current lockfile, independent of PR activity. This reduces Detection from 7 to 2 (automated detection), dropping RPN from 280 to 80 (Major). The D3 comment already documents the gap; the missing piece is the compensating control that makes detection autonomous rather than PR-triggered.

**Post-correction RPN estimate:** 80 (S=8, O=5, D=2) — Major, no longer Critical.

### FM-021-20260313T001: Undocumented Security Ungrouping Intent (Major, highest in E-07)

| Attribute | Value |
|-----------|-------|
| **Severity** | Major |
| **Element** | E-07: Absence of `applies-to: security-updates` in groups |
| **Strategy Step** | Step 2 (Failure Mode lens: Ambiguous / Missing) |

**Evidence:**
The `dependabot.yml` D4 comment explains: "Security updates are ungrouped by default. Each CVE gets its own PR with clear visibility into what vulnerability it addresses." The comment further notes: "If we later want to GROUP security updates, add a group with `applies-to: security-updates` inside the relevant ecosystem block."

There is no explicit guard, assertion, or documentation in the groups blocks themselves stating that security updates must not be grouped. The phrase "if we later want to GROUP security updates" normalizes grouping as a future option without documenting the RPN-140 failure mode that makes accidental grouping dangerous.

**Analysis:**
Severity=7: Accidental grouping of a security update with a routine version update could delay a CVE fix if the grouped PR fails CI due to an unrelated dep conflict. The risk analysis documents this as RPN 140: "A security update for a transitive dep... grouped with a routine pytest minor bump would obscure whether the CI failure was caused by the security patch or the routine update." Occurrence=4: Future maintainers expanding grouping may not read D4 carefully. Detection=7: There is no automated check preventing a `applies-to: security-updates` key from being added to a groups block.

**Recommendation:**
Add a comment immediately before each `groups:` block in both ecosystems:

```yaml
# SECURITY POLICY: Security updates are intentionally NOT included in any group.
# Dependabot ungrouped security updates by default (correct behavior).
# DO NOT add `applies-to: security-updates` to any group entry.
# Rationale: RPN-140 failure mode — grouped security PRs obscure transitive conflicts,
# delaying CVE fixes. See: projects/PROJ-030-bugs/research/dependabot-risk-analysis.md
```

**Post-correction RPN estimate:** 42 (S=7, O=2, D=3) — Minor, with the comment serving as a deterrent and reference.

### FM-019-20260313T001: Filter B Coupling Without Enforcement (Major)

| Attribute | Value |
|-----------|-------|
| **Severity** | Major |
| **Element** | E-06: `commit-message: prefix: deps/ci` |
| **Strategy Step** | Step 2 (Failure Mode lens: Inconsistent) |

**Evidence:**
The D5 comment states: "Both prefixes are caught by version-bump.yml Filter B (#187) which skips the version-bump job for ci: and deps: prefixed commits." The `version-bump.yml` filter is documented as a downstream dependency on this prefix value. The D5 comment does not warn that changing the prefix requires updating Filter B.

**Analysis:**
Severity=8: If a maintainer changes `prefix: "deps"` to `prefix: "chore(deps)"` (a common conventional-commits style) without updating Filter B, the version-bump job will execute on every Dependabot PR, generating spurious version bump commits. This would produce a feedback loop: Dependabot opens PR → version-bump job triggers → spurious commit is pushed → further automation may trigger. Occurrence=3: Prefix changes are infrequent but plausible during conventional-commits adoption or tooling standardization. Detection=8: Without an explicit cross-reference comment, the coupling is invisible; Filter B is in a separate file with no back-reference to `dependabot.yml`.

**Recommendation:**
Update the D5 comment to:

```yaml
# D5: Commit message prefix
#   - "ci" for actions, "deps" for pip. Unchanged from current config.
#   - Both prefixes are caught by version-bump.yml Filter B (#187) which
#     skips the version-bump job for ci: and deps: prefixed commits.
#   - WARNING: Changing either prefix REQUIRES updating Filter B in
#     .github/workflows/version-bump.yml simultaneously. Failure to do so
#     will cause the version-bump job to run on Dependabot PRs, producing
#     spurious version bump commits. See version-bump.yml lines matching
#     "ci\|deps" for the filter expression.
```

**Post-correction RPN estimate:** 48 (S=8, O=3, D=2) — Minor, with cross-reference reducing detection difficulty from 8 to 2.

### FM-013-20260313T001: Security PR Slot Competition (Major)

| Attribute | Value |
|-----------|-------|
| **Severity** | Major |
| **Element** | E-04: `open-pull-requests-limit: 10` |
| **Strategy Step** | Step 2 (Failure Mode lens: Insufficient) |

**Evidence:**
The D4 comment notes: "Note: `open-pull-requests-limit` applies to security PRs too (CC-005). At Jerry's scale this is unlikely to saturate, but a burst of CVEs could compete with version update PRs for queue slots."

The risk analysis (Section 4, Recommendation R-6) states: "Security updates always individual — add `open-pull-requests-limit: 0` override or separate security block."

**Analysis:**
Severity=7: If the limit saturates during a CVE burst, Dependabot stops opening security PRs silently — there is no notification that new CVE PRs were suppressed. Occurrence=3: CVE bursts are uncommon but occur (e.g., Log4Shell class events affecting transitive deps). Detection=5: The suppression is silent; only a manual review of Dependabot's PR activity log would reveal that expected security PRs are missing.

**Recommendation:**
GitHub Dependabot does not currently support a separate `open-pull-requests-limit` for security updates vs. version updates at the ecosystem level. The compensating control is monitoring: add a D6 comment note that during a CVE burst, close pending version update PRs to free slots for security PRs. Additionally, document the recommended response in an incident runbook: "If a known CVE has no Dependabot PR after 24 hours, check if the PR limit is saturated."

**Post-correction RPN estimate:** 56 (S=7, O=2, D=4) — Minor.

### FM-016-20260313T001: Weekly Schedule vs. Non-Security Vulnerability Window (Major)

| Attribute | Value |
|-----------|-------|
| **Severity** | Major |
| **Element** | E-05: `schedule: interval: weekly, day: monday` |
| **Strategy Step** | Step 2 (Failure Mode lens: Insufficient) |

**Evidence:**
The `dependabot.yml` D4 comment states: "Dependabot security updates are event-driven (triggered by GitHub Advisory Database updates), not schedule-driven." The risk analysis (Section 4) confirms security updates are individual and event-driven. However, the schedule governs version updates — including minor bumps of packages where a known-but-not-yet-advisory-listed vulnerability exists.

**Analysis:**
Severity=5: A vulnerability disclosed in a package's changelog but not yet in the GitHub Advisory Database will not trigger a Dependabot security PR. The next version update PR (up to 7 days away for a Monday schedule) is the earliest Dependabot-triggered exposure. The `pip-audit` CI step (which runs on every PR) is the compensating control. Occurrence=4: Pre-advisory vulnerability windows are well-documented in open source security practice. Detection=5: pip-audit on CI provides detection when a PR is open, but there is no autonomous scheduled scan of the lockfile between PRs.

**Recommendation:**
Document the compensating control chain explicitly in D5 or a dedicated "Security Detection Controls" comment: "Detection chain for non-advisory vulnerabilities: (1) pip-audit on every CI run catches listed CVEs; (2) weekly version update PR triggers pip-audit; (3) for unlisted vulnerabilities, subscribe to package changelogs (jsonschema, pyyaml, tiktoken, jsonschema have active security mailing lists)."

**Post-correction RPN estimate:** 60 (S=5, O=4, D=3) — Minor.
