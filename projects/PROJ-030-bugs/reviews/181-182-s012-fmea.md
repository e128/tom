## Summary

The corrected `settings.local.json` (post-issues #181/#182) represents a significant improvement over its predecessor: removing 31 deprecated `:*` Bash entries, eliminating redundant MCP specifics, and covering all 19 registered skills. However, six distinct failure domains remain across the file's four logical sections (Skills, MCP, Bash, Hooks). The highest-risk finding is **FM-003** (skill name `user-experience` maps to a meta-skill directory that may not resolve as a simple `Skill()` permission, RPN 420) and **FM-001** (no version-pinning or schema $ref in a file with precedence 3 over committed settings, RPN 336). Eight failure modes are Critical (RPN ≥ 200) and three are Major (RPN 80–199). The file requires **targeted corrections** before the configuration can be considered robustly governed. Overall assessment: **REVISE** — not a full rejection, but Critical findings must be addressed.

## Finding Details

### FM-001-20260314: No `$schema` Reference

| Attribute | Value |
|-----------|-------|
| **Severity** | Critical (RPN 336) |
| **Element** | EL-01 (File-Level Structure) |
| **Strategy Step** | Step 2 — Missing lens |

**Evidence:**
`settings.local.json` line 1 begins with `{` directly; no `$schema` field present. The committed `settings.json` includes `"$schema": "https://json.schemastore.org/claude-code-settings.json"` at line 2. The local file was fixed in issue #180 to use correct field names (`allow` instead of `allowed_tools`) — but the schema anchor that enables IDE validation and automated schema-checking was not added.

**Analysis:**
Without `$schema`, editors cannot validate field names, array types, or value constraints against the canonical Claude Code schema. Any future edits to `settings.local.json` (e.g., adding an `ask` array, adding a hook) happen without schema guard-rails. The schema is the machine-readable contract for this file; its absence makes all FINDING-001-class bugs (wrong field names) recur silently.

**Recommendation:**
Add `"$schema": "https://json.schemastore.org/claude-code-settings.json"` as the first field in the JSON object, matching the pattern in `settings.json`.

**Post-Correction RPN:** S=7, O=2, D=3 = **42** (detection drops to 3 with schema validation active)

### FM-002-20260314: settings.json/local Interaction Not Documented

| Attribute | Value |
|-----------|-------|
| **Severity** | Critical (RPN 210) |
| **Element** | EL-01 (File-Level Structure) |
| **Strategy Step** | Step 2 — Ambiguous lens |

**Evidence:**
The design document (`settings-local-json-design.md`, Decision 4) explicitly analyzed `git push` and determined the local `allow` intentionally overrides the shared `ask`. However, the current `settings.local.json` does NOT contain `Bash(git push *)` in `allow` — this entry was removed from the final file. The design document at Decision 4 says "Keep it in local `allow`" but the implemented file does not include it. This discrepancy means the actual behavior (whether `git push` requires approval or not) is determined solely by `settings.json` line 38 (`"Bash(git push *)"` in `ask`), and the reasoning is not traceable from the local file.

**Analysis:**
Any developer reading `settings.local.json` in isolation cannot determine which permissions are inherited from `settings.json` and which are locally overridden. The interaction semantics (precedence 3 overrides precedence 4; deny wins regardless) are critical to understanding the effective permission set but are invisible from the local file alone.

**Recommendation:**
Add a top-level comment field (JSON does not support comments natively, but a `"_comment"` key is a common convention) or create a companion `SETTINGS-LOCAL-DESIGN.md` in `.claude/` with the 5-decision rationale. Alternatively, use the existing design document and add a cross-reference path to it from a README in `.claude/`.

**Post-Correction RPN:** S=6, O=2, D=4 = **48**

### FM-003-20260314: `Skill(user-experience)` May Not Resolve

| Attribute | Value |
|-----------|-------|
| **Severity** | Critical (RPN 420) |
| **Element** | EL-02 (Skill Entries) |
| **Strategy Step** | Step 2 — Missing/Insufficient lens |

**Evidence:**
`settings.local.json` line 22: `"Skill(user-experience)"`. The `user-experience` skill is a meta-skill with 11 sub-skills in separate directories (`skills/ux-ai-first-design/SKILL.md`, `skills/ux-atomic-design/SKILL.md`, etc.). The Claude Code skill permission documentation specifies `Skill(name)` as an exact match against a skill's registered name. The sub-skills have names `ux-ai-first-design`, `ux-atomic-design`, etc. — not `user-experience`. The design document (Decision 2) notes "The 10 `ux-*` sub-skills are sub-skills of `/user-experience` and are invoked through the parent skill, not independently permissioned" — but this assumption is unverified.

**Analysis:**
If Claude Code routes sub-skill invocations through the `user-experience` parent, then `Skill(user-experience)` is sufficient. If sub-skills are independently invocable (each with their own `Skill()` identity), the 11 sub-skill permissions are missing, and any direct invocation of a UX sub-skill will trigger an approval prompt. Given the `/user-experience` skill has 11 specialist agents (ux-orchestrator + 10 specialists) and the trigger map routes to `/user-experience` as a unit, the parent-skill routing assumption is plausible but unconfirmed.

**Recommendation:**
Test: invoke `/user-experience` and check whether any sub-skill invocations trigger approval prompts. If sub-skill approval prompts appear, add `Skill(ux-heuristic-eval)`, `Skill(ux-heart-metrics)`, `Skill(ux-atomic-design)`, `Skill(ux-lean-ux)`, `Skill(ux-jtbd)`, `Skill(ux-kano-model)`, `Skill(ux-behavior-design)`, `Skill(ux-design-sprint)`, `Skill(ux-ai-first-design)`, `Skill(ux-inclusive-design)`, `Skill(ux-lean-ux)` to the allow array.

### FM-005-20260314: Skill Name Drift Risk

| Attribute | Value |
|-----------|-------|
| **Severity** | Critical (RPN 216) |
| **Element** | EL-02 (Skill Entries) |
| **Strategy Step** | Step 2 — Inconsistent lens |

**Evidence:**
`Skill(name)` matching is exact and case-sensitive (per `docs/reference/claude-code-permissions.md` Skill Permissions section). The 19 skill names in `settings.local.json` are manually maintained string literals with no automated verification. The git history shows skill names have changed (e.g., `user-experience` skill was added relatively recently as a meta-skill consolidating the 11 `ux-*` sub-skills). There is no CI gate that validates `Skill()` entries against the actual `skills/*/SKILL.md` `name:` field values.

**Analysis:**
Any skill rename, addition, or removal requires a parallel manual update to `settings.local.json`. This file is gitignored, so CI cannot catch stale entries. The silent failure mode (stale permission entry → approval prompt rather than error) means the breakage is low-visibility and occurs at runtime during active work sessions.

**Recommendation:**
Add a CI or pre-commit validation script that reads `skills/*/SKILL.md` frontmatter `name:` fields and cross-references them against `settings.local.json` `Skill()` entries. This is a best-effort check since `settings.local.json` is gitignored — the script would need to run locally, not in CI. Document the manual sync requirement in `.claude/README.md` or the design document.

**Post-Correction RPN:** S=6, O=2, D=5 = **60**

### FM-010-20260314: `Bash(git stash *)` Permits Destructive Stash Operations

| Attribute | Value |
|-----------|-------|
| **Severity** | Critical (RPN 210) |
| **Element** | EL-04 (Bash Entries) |
| **Strategy Step** | Step 2 — Insufficient lens |

**Evidence:**
`settings.local.json` line 27: `"Bash(git stash *)"`. The `git stash *` pattern matches `git stash pop`, `git stash drop`, `git stash clear`, and `git stash branch`. The `git stash drop` and `git stash clear` commands permanently delete stashed work. The committed `settings.json` places `Bash(git reset *)` in the `ask` array as a destructive-op gate; stash destructive operations have no equivalent gate.

**Analysis:**
Claude Code agents can execute `git stash drop stash@{0}` or `git stash clear` without user approval, destroying uncommitted work that the developer stashed. While this requires the agent to choose to run such a command (which is unlikely in normal operation), the permission configuration should not silently pre-approve irreversible destructive operations. The `git stash push` operation (which is safe and the primary use case) could be permitted separately.

**Recommendation:**
Replace `"Bash(git stash *)"` with `"Bash(git stash list *)"` and `"Bash(git stash push *)"` in `allow`. Add `"Bash(git stash pop *)"` and `"Bash(git stash drop *)"` to `ask` (noting that settings.local.json currently has no `ask` array — add one with these entries). OR: keep the broad form and add an explicit accepted-risk note in the design document.

**Post-Correction RPN:** S=7, O=2, D=4 = **56**

### FM-013-20260314: PreToolUse Hook Matcher Syntax Unverified

| Attribute | Value |
|-----------|-------|
| **Severity** | Critical (RPN 256) |
| **Element** | EL-05 (PreToolUse Hook) |
| **Strategy Step** | Step 2 — Ambiguous lens |

**Evidence:**
`settings.local.json` lines 34: `"matcher": "WebFetch|WebSearch"`. The pipe `|` is regex alternation syntax. The Claude Code documentation for `PreToolUse` hooks specifies a `matcher` field but does not explicitly state whether it accepts regex or plain string patterns. If Claude Code interprets `matcher` as a plain string, `"WebFetch|WebSearch"` would fail to match either `WebFetch` or `WebSearch` individually — the hook would only fire if the tool name were literally `"WebFetch|WebSearch"`. In that case, both web tools would require manual approval despite the hook's intent to auto-allow them.

**Analysis:**
The design document does not address matcher syntax semantics. If this matcher silently fails (plain-string interpretation), the developer would see approval prompts for WebSearch and WebFetch every session. The high Detection rating (8) reflects that matcher behavior is not observable without triggering the tools.

**Recommendation:**
Verify matcher regex support by testing: invoke a WebSearch tool and observe whether the hook fires. If regex is not supported, split into two hooks: one with `"matcher": "WebFetch"` and one with `"matcher": "WebSearch"`. Alternatively, add a glob-style `"matcher": "Web*"` if Claude Code supports glob matching.

**Post-Correction RPN:** S=8, O=2, D=4 = **64**

### FM-014-20260314: PreToolUse Hook Auto-Allows Without Audit Trail

| Attribute | Value |
|-----------|-------|
| **Severity** | Critical (RPN 245) |
| **Element** | EL-05 (PreToolUse Hook) |
| **Strategy Step** | Step 2 — Insufficient lens |

**Evidence:**
`settings.local.json` lines 35–40: `"command": "echo '{\"decision\":\"allow\"}'", "timeout": 5`. The hook unconditionally emits `{"decision":"allow"}` for any WebFetch or WebSearch invocation. There is no logging, no URL inspection, and no conditional logic. The design document's threat model (T-04) notes that MCP wildcard scope is an accepted risk — but there is no equivalent accepted-risk documentation for unrestricted web access via hooks.

**Analysis:**
WebFetch and WebSearch auto-allow means Claude Code agents can fetch arbitrary URLs without user awareness. While this is a developer convenience feature, it is a security consideration when the developer is not watching the terminal (e.g., background agent execution with `"background": true`). The absence of any audit trail means there is no record of what URLs were fetched or what searches were performed during a session.

**Recommendation:**
Replace the bare echo with a logging command: `echo "Claude web access: $CLAUDE_TOOL_INPUT" >> ~/.claude/web-access.log && echo '{"decision":"allow"}'`. This adds an audit trail with negligible performance overhead. If `$CLAUDE_TOOL_INPUT` is not available in the hook environment, substitute the tool name or timestamp.

**Post-Correction RPN:** S=5, O=5, D=4 = **100**
