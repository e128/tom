## Steelman Reconstruction

### Design Decision 1: `uv sync` → `uv sync --frozen` [SM-001] [SM-002]

**Reconstructed argument:**

The `--frozen` flag is the only semantically correct choice for CI dependency installation. Without `--frozen`, `uv sync` performs a full dependency re-resolution using the installed Python version's metadata — a behavior designed for development environments where the lockfile may need updating. In a CI pipeline, this behavior is a footgun: it silently mutates `uv.lock` whenever the resolver produces a different solution (which can happen across Python minor versions, including the transition to 3.14 where resolver behaviour changed). [SM-001]

The `--frozen` flag enforces the invariant that CI installs exactly what was committed. This invariant has three compounding benefits:

1. **Reproducibility:** Every CI run installs an identical dependency graph, eliminating "works on my machine" failures caused by resolution drift.
2. **Security:** The lockfile is the dependency integrity manifest. Allowing CI to mutate it means a compromised upstream package could alter the lockfile mid-run without the mutation being reviewed by a human.
3. **Early failure signal:** If `uv.lock` is genuinely stale (e.g., after adding a new dependency without running `uv lock`), `--frozen` causes an immediate, clear failure at the install step — far better than a cryptic downstream failure when `bump-my-version` refuses to commit against a dirty tree. [SM-002]

The fix applies `--frozen` in both the initial install step (line 83) and the post-bump validation step (line 156). The second application is equally important: after `bump-my-version` updates `pyproject.toml`'s version field, a non-frozen `uv sync` would attempt a re-resolution incorporating the new version string, risking an unnecessary `uv.lock` mutation during the validation phase.

**Best case conditions:** Python resolver behaviour remains stable within a major version; the project's lockfile is kept current by developer discipline (running `uv lock` after dependency changes).

### Design Decision 2: Fail-Fast "Ensure Clean Working Tree" Step [SM-003] [SM-004]

**Reconstructed argument:**

The clean-tree check (lines 113–123) implements the **fail-fast principle** for CI pipeline reliability. There are two philosophically different responses to an unexpectedly dirty tree: silent recovery (e.g., `git checkout -- .`) or loud failure. The fix deliberately chooses loud failure, and this is the stronger design. [SM-003]

Silent recovery would mask the root cause. If the tree is dirty despite `--frozen`, the cause is unknown — it could be a tool misconfiguration, a file generated by a test fixture, a future dependency version that writes to project directories, or an environment side effect. Silently resetting these changes means the pipeline succeeds while the underlying anomaly goes uninvestigated, potentially compounding over time.

By contrast, the `::error::` annotation (line 117) surfaces the anomaly in the GitHub Actions UI with a specific diagnostic message. The accompanying `git diff --name-only` output (line 118) provides the exact files affected, giving the operator the information needed to diagnose and fix the root cause. The inline guidance (lines 120–122) distinguishes the most likely cause (`uv.lock` needing regeneration) and prescribes the correct remediation (`uv lock` locally + commit). [SM-004]

This step also functions as a correctness guarantee for `bump-my-version`: the tool's `allow_dirty = false` configuration is a safety net ensuring version bump commits are clean. The explicit pre-check converts `bump-my-version`'s opaque failure ("working tree not clean") into a diagnostic failure with context, reducing mean time to resolution.

**Best case conditions:** The `--frozen` fix is the primary defence; this step is the secondary defence. Together, they implement defence-in-depth for working-tree hygiene.

### Design Decision 3: Skip Detection via `github.actor` [SM-005] [SM-006]

**Reconstructed argument:**

The original `github.event.head_commit.author.name` check was an unreliable identity signal. The `author.name` field is populated from the git commit's author metadata, which is set by `git config user.name` at commit time. In the version bump workflow itself (line 135), the pipeline sets `user.name = "github-actions[bot]"` — but this is a configurable string, not a verified identity. Any developer or automated system can set `git config user.name "github-actions[bot]"` and create a commit that would be misidentified as a bot commit, causing the version bump to be incorrectly skipped. [SM-005]

`github.actor`, by contrast, is the GitHub-authenticated principal that triggered the workflow run. It is set by the GitHub Actions platform from the authenticated token, not from any user-configurable git metadata. It cannot be spoofed by commit content. When the version-bump pipeline commits and pushes, the push event's `github.actor` will be `github-actions[bot]` — the authenticated service account identity, not a user-controlled string. [SM-006]

The security improvement is asymmetric in cost: the change is a single field name replacement with zero runtime cost, while the attack surface it closes — an adversary suppressing version bumps by impersonating the bot identity in git metadata — is a genuine, if low-severity, pipeline manipulation vector. The principle of least surprise also favours `github.actor`: it is the field developers expect to represent "who triggered this run," and its semantics are documented and stable in the GitHub Actions specification.

**Best case conditions:** The repository continues using standard GitHub Actions authentication. Non-standard setups using GitHub App tokens with custom actor identities would need to update the exclusion value accordingly.

### Design Decision 4: Catch-Up Tag Script with Confirmation Guard and Provenance Warning [SM-007] [SM-008] [SM-009]

**Reconstructed argument:**

The catch-up tag script (`create-catchup-tags.sh`) addresses the 36-tag deficit through a graduated execution model that makes destructive operations opt-in at each escalation level. [SM-007]

The three-mode design (dry run → `--apply` → `--push`) is the strongest available architecture for a recovery script that:
- Must be auditable (operator needs to verify the tag list before acting)
- Carries irreversible side effects (`--push` triggers `release.yml` for each tag)
- May be run by operators with varying familiarity with the codebase

Dry run as the default (`MODE="${1:-}"`, no argument = dry run) means accidental execution without arguments produces zero side effects. This is the safe failure mode for a script that touches release infrastructure. [SM-007]

The interactive confirmation prompt before `--push` (line 82: `read -p "Continue pushing N tags? (yes/no)"`) follows the principle that destructive bulk operations require explicit human acknowledgement. The `yes/no` prompt (not `y/n`) requires deliberate typing, reducing the likelihood of an accidental confirmation from a muscle-memory `y` keypress. [SM-008]

The provenance warning on lines 78–80 is a forward-looking correctness statement. The annotated tags created by this script lack GPG signatures that would normally be present if the release pipeline had been functioning. Documenting this limitation inline — at the point of use, not in a distant README — ensures operators have the information they need to make an informed decision about the provenance implications of the catch-up tags. [SM-009]

The script's commit-matching logic (grep on `"Bump version:"`, extraction of version string via sed, existence check via `git tag -l`) is minimal and idempotent: running the script twice produces the same result because the existence check skips already-created tags.

**Best case conditions:** The local git clone has full history (`fetch-depth: 0` equivalent). All 36 version bump commits follow the standard `Bump version: X → Y` message format (which they do, as this format is generated by `bump-my-version` itself).

## Best Case Scenario

The BUG-003 fix is strongest under the following conditions — all of which hold in the Jerry repository:

1. **`uv.lock` is kept current.** The `--frozen` flag is effective only when developers run `uv lock` after dependency changes. The project's existing discipline (lockfile committed to version control) supports this.

2. **Python 3.14 is the only version installed in CI.** The root cause (resolver behaviour change) is a per-Python-version phenomenon. The fix eliminates the class of failure rather than patching the specific 3.14 manifestation, so it is robust to future Python version upgrades.

3. **`bump-my-version` retains `allow_dirty = false`.** The fail-fast clean-tree check is architecturally paired with this configuration. If `allow_dirty` were ever changed to `true`, the check would become redundant but harmless.

4. **GitHub Actions token authentication is used.** The `github.actor` check is reliable as long as the repository uses standard GitHub Actions authentication. Non-standard app token setups may need to update the actor exclusion value.

5. **The 36 catch-up tags are pushed before any operator assumes tag = released version.** The script's graduated execution model makes this safe; the interactive confirmation ensures the operator understands the release-workflow implications before committing to the push.

**Rational evaluator confidence:** HIGH. All four design decisions follow established CI/CD best practices (lockfile pinning, fail-fast, authenticated identity, graduated destructive operations). The fix is minimal — it changes the fewest possible lines to address each root cause — which limits the surface area for regressions.
